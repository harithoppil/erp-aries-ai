# Stage 4 — HR Module Prisma Port (Leave / Salary / Payroll / Attendance / Appraisal / Job Opening / Shift Type)

> **Goal**: Bring HR depth to parity with ERPNext by porting the ~17 missing HR Prisma models, seeding their `tabDocField` metadata so the existing `ERPFormClient` + `ERPListClient` render them automatically, and wiring submit-time side-effects (payroll → GL, leave → balance) into the document orchestrator.
>
> **Outcome at end of stage**: `/dashboard/erp/leave-application/new`, `/dashboard/erp/salary-slip/new`, `/dashboard/erp/attendance` etc. all render the proper Frappe-style tabbed form, with GL entries posting on Salary Slip submit. Sidebar entries exist. CSV import for Employee + Holiday List works.
>
> **Effort**: 3–4 days (longest of the four stages because it touches the schema, controllers, and data).
>
> **Depends on**: Nothing technically — but practically, do Stages 1–3 first so the HR forms inherit the metadata-driven UX rather than being one-offs.

---

## 0. Why this is harder than the other stages

The ERPNext HR module isn't part of the core `erpnext` app — it lives in the separate `hrms` Frappe app. Neither dump in `sql/` carries the HR tables:
- `sql/aries_site_local.sql` was captured from a Frappe site without HRMS installed.
- `sql/aries_azure_nextjs_post_frappe_migration.sql` was generated by mirroring that same source.

Confirm before starting:
```bash
grep -iE "CREATE TABLE.*[Tt]ab(Leave|Salary|Payroll|Attendance|Appraisal|JobOpening|ShiftType)" sql/aries_site_local.sql
# expect: zero matches
```

So this stage isn't a "copy table from Frappe → load into Postgres". It's a deliberate port from the **upstream HRMS DocType JSON** files (publicly available at `frappe/hrms` on GitHub).

## 2. Source of truth for each DocType definition

Pull from `frappe/hrms` upstream (read-only reference, do not vendor the whole repo):
- Repository: `https://github.com/frappe/hrms` — branch `version-15`
- DocType JSON path: `hrms/hr/doctype/<snake_case>/<snake_case>.json` (e.g. `leave_application/leave_application.json`)
- Each JSON has a `fields[]` array — this is what becomes our Prisma columns AND our `tabDocField` rows.

A small script converts each JSON to the two artefacts. See §4 for the script.

## 3. Files to create

### 3.1 `prisma/schema.prisma` — append HR models
Roughly 250–350 lines of `model X { ... }` blocks. Mechanically follow the same shape as `Employee` (already in schema) so the generated Prisma client treats them identically.

Required common columns (every doctype gets these):
```prisma
model SalarySlip {
  name           String    @id @db.VarChar(140)
  creation       DateTime? @default(now()) @db.Timestamptz(3)
  modified       DateTime? @updatedAt @db.Timestamptz(3)
  modified_by    String?   @db.VarChar(255)
  owner          String?   @db.VarChar(255)
  docstatus      Int       @default(0)
  idx            Int       @default(0)
  // ── domain columns from frappe/hrms JSON ──────────
  employee       String    @db.VarChar(140)
  posting_date   DateTime  @db.Date
  start_date     DateTime  @db.Date
  end_date       DateTime  @db.Date
  total_working_days Float @default(0)
  gross_pay      Float     @default(0)
  net_pay        Float     @default(0)
  total_deduction Float    @default(0)
  // ... rest of columns ...
  @@index([employee])
  @@index([posting_date])
  @@map("salary_slip")
}
```

Child tables follow the same convention as existing Frappe-style children (`parent` / `parentfield` / `parenttype` columns):
```prisma
model SalaryDetail {
  name           String    @id @db.VarChar(140)
  parent         String    @db.VarChar(140)
  parentfield    String    @db.VarChar(140)
  parenttype     String    @db.VarChar(140)
  idx            Int       @default(0)
  salary_component String   @db.VarChar(140)
  amount         Float     @default(0)
  // ...
  @@index([parent])
  @@map("salary_detail")
}
```

### 3.2 `scripts/hr/import-doctype-defs.ts`
Programmatic importer that reads HRMS DocType JSON files (vendored under `scripts/hr/upstream-defs/`) and produces:
1. A `.prisma` snippet per model — appended to `prisma/schema.prisma` (via marker comments `// === HR_PORT_START ===` / `// === HR_PORT_END ===`).
2. An `INSERT INTO "tabDocField" (...) VALUES (...)` SQL script per model — written to `scripts/hr/seed-docfield.sql`.
3. A `prisma db push` invocation at the end (manual: it's destructive on a non-empty schema, only run on empty HR tables).

Shape:
```ts
async function main() {
  for (const file of glob('scripts/hr/upstream-defs/*.json')) {
    const def = JSON.parse(readFileSync(file, 'utf-8'));
    const model = toPrismaModel(def);
    appendToSchema(model);
    const docfields = toDocFieldRows(def);
    appendToSeed(docfields);
  }
}
```

Two helper modules:
- `scripts/hr/json-to-prisma.ts` — `fieldtype` → Prisma type mapping (`Data`→`String`, `Check`→`Boolean`, `Int`→`Int`, `Currency`→`Float`/`Decimal`, `Link`→`String`, `Table`→relation handled separately, etc.).
- `scripts/hr/json-to-docfield.ts` — builds one `tabDocField` row per `field` entry, copying `fieldname`/`label`/`fieldtype`/`options`/`idx`/`reqd`/`hidden`/`read_only`/`in_list_view`/`in_standard_filter`/`depends_on`/etc. Sort by JSON file order to set `idx` correctly.

### 3.4 `lib/erpnext/controllers/hr-leave-application.ts`
Submit-side controller for leave. Pattern mirrors `lib/erpnext/controllers/selling-sales-invoice.ts`:

```ts
export interface LeaveApplicationDoc { /* ... */ }

export function validateLeaveApplication(doc: LeaveApplicationDoc): ValidationResult {
  // 1. Check employee exists, employee not disabled.
  // 2. Check from_date < to_date.
  // 3. Check overlapping leaves for same employee.
  // 4. Check leave balance from LeaveAllocation.
  // 5. Check leave_type allows half-day if doc.half_day.
}

export function onSubmitLeaveApplication(doc: LeaveApplicationDoc): SubmitResult {
  // 1. Compute total_leave_days = business days between from/to.
  // 2. Update LeaveAllocation.leaves_taken (or create Leave Ledger Entry).
  // 3. No GL entries (leave isn't a financial event).
  return { success: true };
}

export function onCancelLeaveApplication(doc: LeaveApplicationDoc): CancelResult {
  // Reverse the LeaveAllocation update.
}
```

### 3.5 `lib/erpnext/controllers/hr-salary-slip.ts`
The big one. Salary Slip submit posts GL entries:

```ts
export function onSubmitSalarySlip(doc: SalarySlipDoc): SubmitResult {
  const glEntries: GlEntryInput[] = [];
  for (const earning of doc.earnings) {
    glEntries.push({
      account: earning.salary_component_account,   // Salary Expense (debit)
      debit: earning.amount, credit: 0,
      voucherType: 'Salary Slip', voucherNo: doc.name,
      // ...
    });
  }
  for (const deduction of doc.deductions) {
    glEntries.push({
      account: deduction.salary_component_account, // Statutory Payable (credit)
      debit: 0, credit: deduction.amount,
      // ...
    });
  }
  // Net pay → Salary Payable account (credit)
  glEntries.push({
    account: doc.payroll_payable_account,
    debit: 0, credit: doc.net_pay,
    // ...
  });
  return { success: true, gl_entries: glEntries };
}
```

## 7. Risks & mitigations

| Risk | Mitigation |
|---|---|
| Azure has a different schema than local — `prisma db push` against Azure could destroy data | **Never run** `db push` against Azure during this stage. Use migrations or generate plain `ALTER TABLE` scripts and review before applying |
| Upstream HRMS DocType JSON changes between Frappe versions | Vendor a specific commit (`version-15`) and pin via a `PROVENANCE.md` in `scripts/hr/upstream-defs/` |
| Some HR fields reference doctypes we DON'T have (e.g. `EmployeeOnboarding` → `OnboardingActivity`) | Either also port the dependency, or change the field to `Data` (string) and document the loss |
| Salary Slip GL posting needs `Cost Center` / `Project` / `Department` defaults to resolve | Implement same fallback chain as Sales Invoice controller (employee.department.cost_center → company.cost_center) |
| Leave balance computation needs a Leave Ledger Entry table (Frappe has it) | Add as a 18th doctype OR compute on-the-fly from `LeaveAllocation - SUM(LeaveApplication.total_leave_days WHERE docstatus=1)`. Pick the cheaper option for now |
| Attendance has uniqueness `(employee, attendance_date)` | Add a unique index `@@unique([employee, attendance_date])` in Prisma |
| `tabDocField` rows from HRMS may conflict with existing ERPNext core docfield rows | Frappe namespaces by `parent` (doctype name), so HRMS doctypes never collide with core. Confirm with `SELECT parent, COUNT(*) FROM "tabDocField" WHERE parent IN (<HR doctypes>) GROUP BY parent` — expect 0 before seed, ~17 after |
| Importing employees in bulk fails because `Department` master is empty | Stage 4 seed includes a default Department + Designation + Branch row so imports satisfy the FK by default |
